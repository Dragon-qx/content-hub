## 1. 产品概述

### 1.1 一句话定位

**ContentHub** 是面向内容从业者与团队的工业级全自动化内容管理平台，统一管理多社交账号的内容创作、发布、数据监测和分析，实现从内容生产到数据复盘的全链路数字化。

### 1.2 市场背景

当前内容运营面临的核心痛点：

| 痛点 | 说明 |
|------|------|
| **账号碎片化** | 平均每个运营要管 5-10 个平台账号，来回切换耗时 |
| **发布效率低** | 手动逐平台发布，重复操作多，容易遗漏 |
| **数据割裂** | 各平台数据不互通，没有统一的对比视角 |
| **协作混乱** | 内容审核靠微信/Excel，版本管理混乱 |
| **合规风险** | 操作不可追溯，出问题无据可查 |

### 1.3 竞品分析

| 产品 | 定位 | 优势 | 不足 |
|------|------|------|------|
| **Buffer** | 海外社交平台调度 | UI 简洁、定时发布成熟 | 国内平台不支持、无审批流 |
| **Hootsuite** | 企业级社交管理 | 品牌监控强、报告完善 | 价格高、国内平台缺失 |
| **蚁小二** | 国内多平台分发 | 国内平台覆盖全 | UI 老旧、数据分析弱 |
| **新榜** | 内容数据+分发 | 榜单数据权威 | 偏监测、轻管理 |
| **ContentHub (我们)** | 国内为主+海外覆盖 | 审批流、全链路、团队协作 | 新品、需时间沉淀 |

### 1.4 核心价值主张

> **"让内容团队把时间花在创作上，而不是操作上。"**

- **效率提升 10 倍**：一次编辑，多平台一键分发
- **数据驱动决策**：统一指标、趋势分析、异常告警
- **团队协作无忧**：审批流、权限管控、版本管理
- **合规可审计**：全操作日志，可追溯可审计

## 2. 目标用户

### 2.1 用户画像

#### 🎯 运营专家 — 小林，26 岁

- **背景**：某 MCN 运营专员，管理 12 个抖音账号、8 个小红书账号
- **日常**：每天要发 30+ 篇内容，手动逐平台上传
- **痛点**：效率低、容易出错、加班多
- **需求**：批量发布、定时排期、多账号切换方便

#### 🎯 内容主编 — 张主编，35 岁

- **背景**：某品牌内容编辑部主编，带 5 人小编团队
- **日常**：审核小编产出的内容、把控质量、安排排期
- **痛点**：审核靠微信截图、版本混乱、责任不清
- **需求**：审批流程、版本管理、团队看板、绩效统计

#### 🎯 自媒体老板 — 王总，32 岁

- **背景**：创办了一个小型传媒公司，30+ 达人
- **日常**：统筹所有达人和品牌账号，对接甲方
- **痛点**：不知道各账号真实数据、内容质量参差不齐
- **需求**：统一看板、达人管理、数据报表、客户报告

### 2.2 用户旅程

```
发现账号和内容机会 → 创建/编辑内容 → 提交审批 → 审核通过 → 排期发布
     ↓                      ↓              ↓          ↓         ↓
  热点追踪              编辑器+AI辅助     通知主编   多级审批   定时/实时
                                                              ↓
                                                    发布成功/失败通知
                                                              ↓
                                                    数据复盘 → 指导下次创作
```

#### 内容状态机

```
DRAFT → IN_REVIEW → APPROVED → SCHEDULED → PUBLISHING → PUBLISHED
  ↓         ↓           ↓          ↓           ↓
  └→ EDIT  └→ REJECT  └→ EDIT   └→ CANCEL   └→ FAILED → RETRY
```

| 状态 | 含义 | 可执行操作 |
|------|------|-----------|
| DRAFT | 草稿 | 编辑、提交审核、删除 |
| IN_REVIEW | 审核中 | 查看、撤回、审批/驳回 |
| APPROVED | 审核通过 | 排期发布、编辑 |
| SCHEDULED | 已排期 | 取消发布、编辑 |
| PUBLISHING | 发布中 | 查看进度 |
| PUBLISHED | 已发布 | 查看数据、归档 |
| FAILED | 发布失败 | 重试、编辑 |
| ARCHIVED | 已归档 | 查看历史 |

## 4. 非功能性需求

### 4.1 性能

| 指标 | 目标 | 测试方法 |
|------|------|---------|
| API P99 延迟 | ≤ 200ms | k6 压测 |
| 数据查询 P95 | ≤ 500ms | k6 压测 |
| 并发用户 | ≥ 1,000 | 模拟并发 |
| 发布调度精度 | ± 1 秒 | 日志时间戳 |
| 数据更新延迟 | ≤ 5 分钟 | 日志监控 |
| 单文件上传 | ≤ 2GB | 大文件测试 |
| 首屏加载 | ≤ 3s | Lighthouse |

### 4.2 可用性 & 稳定性

- **系统可用性**：≥ 99.9%（月度停机 < 43 分钟）
- **MTTR**（平均恢复时间）< 30 分钟
- **MTBF**（平均无故障时间）> 720 小时
- 滚动更新：零停机部署
- 数据库主从 + 读写分离
- Redis 集群（主从 + Sentinel）
- 多副本部署：API ≥ 2 副本、Worker ≥ 2 副本

### 4.3 安全

| 层面 | 措施 | 优先级 |
|------|------|--------|
| 认证 | JWT + Refresh Token（15min 短期 token） | P0 |
| 授权 | RBAC + 资源级隔离 | P0 |
| 传输 | 全站 TLS 1.3、HSTS | P0 |
| 密码 | argon2id 哈希 | P0 |
| API Key | AES-256-GCM 加密存储 | P0 |
| 输入 | 参数校验 + SQL 注入防护 + XSS 过滤 | P0 |
| 限流 | Redis 滑动窗口，按用户+接口维度 | P0 |
| 审计 | 全操作日志，保留 180 天 | P0 |
| MFA | TOTP 二次认证 | P1 |
| WAF | Web 应用防火墙（生产环境） | P1 |

### 4.4 可扩展性

- **水平扩展**：API 和 Worker 支持水平扩容
- **插件化平台接入**：新平台接入只需实现 PlatformAdapter 接口
- **消息队列解耦**：核心流程通过 BullMQ 消息驱动
- **多租户**：Team 级别数据隔离

### 4.5 兼容性

| 维度 | 要求 |
|------|------|
| 浏览器 | Chrome 90+、Firefox 88+、Safari 14+、Edge 90+ |
| 移动端 | 响应式（V1.0），独立 App（未来） |
| API | 向前兼容（v1 至少支持 12 个月） |

### 4.6 数据合规

| 要求 | 说明 |
|------|------|
| 数据加密存储 | 所有敏感字段加密（API Key、Token） |
| 数据备份 | 每日全量 + 实时 binlog 备份 |
| 数据导出 | 用户可导出自己的全部数据 |
| 数据删除 | 用户注销后 30 天内完全清除数据 |
| 数据隔离 | Team 级别严格隔离 |
| 隐私政策 | 符合 GDPR / 个人信息保护法 |

## 8. 上线策略 & 里程碑

### 8.1 开发里程碑

| 阶段 | 内容 | 时间 | 产出 |
|------|------|------|------|
| **M0** | 脚手架 + CI/CD + 部署 | W1 | 可运行的空项目 |
| **M1** | 用户 + 团队 + 账号绑定 | W2 | 能注册、能绑账号 |
| **M2** | 内容编辑 + 媒体管理 | W3-W4 | 能创作内容 |

| **M5** | 集成测试 + Bug 修复 | W9 | 能交付内测 |
| **内测** | 邀请 10 个种子用户 | W10-W11 | 收集反馈 |

### 8.2 Go/No-Go 决策点

| 节点 | 判断条件 |
|------|---------|
| M2 → M3 | 编辑器无明显卡顿，能上传 50MB 视频 |
| M3 → M4 | 发布成功率 > 95%（测试账号） |
| M4 → M5 | 数据延迟 < 5 分钟 |
| M5 内测 | 无 P0/P1 级别的 critical bug |
